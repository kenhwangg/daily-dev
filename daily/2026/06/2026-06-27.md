# Day 4

## 에이전트 아키텍처의 로그 중심(Log-centric) 구조 관련 확습

* 문제: 에이전트의 작업 로그를 프로세스 기반으로 유지할지, 영속적인 이벤트 로그 기반으로 관리할지에 대한 고민과 학습  
* 검토 대상

  * **기존 프로세스 기반 관리 (Sticky Session)**

    * 에이전트가 실행되는 동안 서버 메모리에 세션 상태를 계속 유지하는 방식
    * 구현이 직관적이나, 수시로 일어나는 배포나 장애 발생 시 실행 중이던 에이전트 맥락이 증발함
    * 대규모 동시 요청 발생 시 서버의 메모리와 커넥션 유지를 위한 인프라 비용이 기하급수적으로 증가
  * **영속적 이벤트 로그 기반 관리 (Log-centric)**

    * 사용자의 입력, 모델 출력, 툴 호출 등 모든 상태 변화를 Append-only 로그로 저장소에 기록하는 방식
    * 에이전트를 구동하는 워커(Worker)를 무상태(Stateless)로 만들 수 있어, 서버가 배포로 인해 죽더라도 새 워커가 로그를 읽어 직전 상태를 즉시 복구(Failover)함
    * 런타임 환경, LLM 벤더(Claude/GPT 등)의 교체가 자유로워 종속성(Lock-in)이 제거되고 디버깅 시 특정 시점의 재현(Replay)이 가능해짐
    * 단, 매 턴마다 날것(Raw)의 데이터를 기록하고 상태를 재구축하는 과정에서 스토리지 I/O 비용 및 비용 폭탄의 리스크 존재

* 결론: 로그 중심 아키텍처의 강력한 신뢰성과 확장성 이점을 취하되, 대규모 환경에서의 비용 문제를 해결하기 위해 최적화 전략을 결합하여 도입하는 방향도 있음
  * **Hot/Cold 티어링**: 활성화된 세션 로그는 Redis/DynamoDB에, 종료된 로그는 즉시 객체 스토리지로 이관하여 스토리지 비용 통제
  * **체크포인팅(Checkpointing)**: 대화가 길어질 경우 주기적으로 최종 상태를 스냅샷으로 저장하여 네트워크 I/O 및 재구축 연산 비용 최소화
  * **보존 정책(TTL)**: 디버깅용 Raw 로그는 14일~30일 후 자동 삭제되도록 설정하여 장기 저장 데이터 용량 관리
