# Day 2

## 리소스 기반 모듈 책임 구조 재검토

* 문제: 리소스별 담당 모듈을 분리하고 API 호출 및 시그널 기반 캐싱 구조를 적용하려고 함
* 검토 결과: 기획 변경에 따라 리소스의 소유권과 영향 범위가 변경되면서 모듈 책임도 함께 이동하는 상황 발생
* 결론: 리소스 중심 설계보다 화면(엔드포인트) 중심 설계가 더 적합하다고 판단
  * 각 화면이 필요한 엔드포인트를 직접 책임지는 구조로 변경
  * 기획 변경 시 모듈 간 영향도 감소
  * API 구조 수정 비용 최소화

## 화면 분석 방식 단순화

* 문제: 화면 분석을 위해 로컬 브라우저 익스텐션 도입을 검토함
* 검토 결과: 화면 코드 접근 권한 요청으로 필요한 분석이 가능함을 확인
* 결론: 별도 익스텐션 도입은 불필요
  * 설치 및 배포 과정 제거
  * 운영 복잡도 감소
  * 유지보수 비용 절감

## 설계 검증 원칙

* 문제: 구현 전 다양한 구조를 선제적으로 설계하는 경향이 있음
* 결론: 설계보다 먼저 요구사항 충족 여부를 검증하는 과정이 필요

  * 불필요한 기능 개발 방지
  * 요구사항 변경 대응 비용 감소
  * 시스템 복잡도 최소화
